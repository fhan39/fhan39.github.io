关于REST API 误用检测与自动修复这篇论文第一部分以及第二部分的归纳总结，同时也梳理了 REST API 的基础组成部分与误用问题的细节，记录如下。

🧭 一、问题提起：REST API 的“误用”是个什么问题？

REST API 是软件开发中最常见的通信接口（Application Programm Interface）。但 REST API 的使用并没有编译器校验支持，开发者容易犯下各种“错误”，比如：

  • 请求路径拼错
  • method 写错（GET 写成 POST）
  • Body 格式错误
  • 忘记传 required 字段
  • 忽略 Header 信息（如 Authorization）

这些误用往往不会直接报错，而是接口返回 400 或 500，影响难以排查。
排查错误过程中，开发者往往陷入反复修改request与查看报错信息的無駄遣い时间中。
因此如果可以通过工具实现自动修复错误的REST API调用，那也许可以进一步提升工程上的开发效率了

🔍 二、什么是 REST API 的三要素？

REST API 本质是 HTTP 协议的一种应用方式，具有以下三要素：

要素 说明 示例
Endpoint 请求路径，标识资源 /v1.1/devices
Method 操作方式（GET、POST 等） POST
Request Data 包括 Header 和 Body Authorization, JSON数据等

🤖 三、如何自动检测并修复误用的提案方法

论文标题为：

Automated Program Repair Based on REST API Specifications Using Large Language Models

该研究提出一种方法：

  1. 识别 API 误用点：通过比较代码调用与 OpenAPI 规范，找出偏离点
  2. 构造 Prompt：将代码片段 + API 规范片段拼接为 Prompt
  3. 输入 LLM（如 GPT-4）：由模型生成修复代码
  4. 验证：判断生成结果是否可用，比较修复率

🧠 四、REST API 与传统语言 API 有何不同？

对比项 Java API REST API
检查机制 编译器校验 无
调用形式 方法调用 HTTP 请求
参数传递 类型系统管理 JSON / URL 约定
IDE 支持 自动补全提示 几乎无

REST API 缺乏类型安全和静态检查，误用情况不易察觉，因此自动修复技术显得尤为必要。

🧵 五、Prompt 是什么？

Prompt 是大语言模型的“输入说明书”。

论文中的 Prompt 并非单纯自然语言，而是结构化内容，包括：

  • 出错的调用代码段
  • OpenAPI 规范片段（YAML 或 JSON）
  • 修复目标的自然语言说明（如“请修复 requestBody 错误”）

Prompt 构造的质量直接决定修复效果。

📌 六、今天的个人收获

  • 第一次从“误用角度”而非“使用角度”学习 REST API
  • 理解了 REST API 的结构与传统语言 API 的根本差异
  • 初步掌握了 OpenAPI 的写法与作用
  • 了解了 prompt engineering 在自动修复中的实践方式
  • 复习了http协议的三要素（计算机网络学完就丢了気がしますww）
