REST API误用修复研究（RAG篇）

Created2025-08-22|Updated2025-08-22|技术笔记毕业研究
|Post Views:

✨ 1. 背景

  • 以往的研究（第一篇论文）利用 AST 语法树 🌳 来检测 client 侧的程序内的 API 调用是否符合规范，并将错误信息作为 prompt 传给 LLM 进行自动修复。
  • 但问题在于：LLM 如果没学过此 API 规范，就可能无法准确修复 ⚠️，存在明显局限性。

🔍 2. RAG 方法的引入

  • RAG (Retrieval-Augmented Generation) 💡 的思路是：在调用 LLM 之前，先检索相关的外部信息，再将其作为上下文补充给 LLM。
  • 在这里，研究者先抓取 REST API 的规范文档 📄，通过使用 “RAG 相关工具” 处理成小片段（chunk），向量化后存入数据库。
  • 修复时，同样将错误代码片段向量化，作为依据输入数据库，检索到相关的 API 规范，再把这些内容和程序一起交给 LLM，从而提升 LLM 修复的准确性。
  • LlamaIndex 🦙 就是上面的 “RAG 相关工具”，负责完成切片、向量化、存储和检索，是实现 RAG 的关键组件。

🗂️ 3. DB 扩展 RAG 方法

  • 在标准 RAG 的基础上，研究者进一步扩展了数据库：
    不仅保存 最新的 REST API 规范 的自然语言和源代码，还将 过去版本的 API 规范 一并存储，总共分为 4类数据库。
  • 这样，当 LLM 面对某些 旧版遗留错误 🕰️ 时，不需要只依赖最新规范，也能从数据库检索到 旧 → 新 的迁移模式，从而提高修复的成功率。

📊 4. 实验结果

  • ⚙️ 仅使用 LLM：修复成功率 60.5%
  • 📚 使用标准 RAG：修复成功率 73.5%
  • 🗃️ 使用 DB 扩展的 RAG：修复成功率 80.0%

👉 可以看到，RAG 显著提高了修复的成功率，尤其在 “规范迁移” 相关的 bug 修复上效果更明显 ✅。

📝 5. 总结

  • 对比来看：
    • 🧩 第一篇论文：提出了“检测 + 修复”的流程，但单纯依赖静态 prompt，LLM 修复仍然吃力，准确性不高。
    • 🚀 第二篇论文：引入 RAG，使得修复更稳定、更精准。
  • 感想 💭：随着 LLM 越来越强大，未来 RAG 的价值可能更多体现在 小众领域 和 新规范的快速适配 上，但仍然具有重要意义。
